Implementação: Service Layer

Regras de negócio e persistência em memória

src/services/tasks.service.js
01let tasks = [];
02let nextId = 1;
03
04const list = () => tasks;
05
06const create = (data) => {
07 const newTask = { id: nextId++, ...data };
08 tasks.push(newTask);
09 return newTask;
10};
11
12const update = (id, data) => {
13 const index = tasks.findIndex(t => t.id === id);
14 if (index === -1) return null;
15 tasks[index] = { ...tasks[index], ...data };
16 return tasks[index];
17};
18
19const remove = (id) => {
20 const index = tasks.findIndex(t => t.id === id);
21 if (index === -1) return false;
22 tasks.splice(index, 1);
23 return true;
24};
25
26module.exports = { list, create, update, remove };

Lógica Aplicada

Persistência

Usamos um Array simples para simular o banco de dados. O nextId garante chaves únicas.

Spread Operator

O uso de ...data permite atualizar apenas os campos enviados, mantendo os outros inalterados (Partial Update).

Isolamento

Note que não existe req ou res aqui. O Service é JavaScript puro, independente do Express.

Dica de Arquitetura

Se quiséssemos trocar o banco de dados (ex: MongoDB), alteraríamos apenas este arquivo. O Controller nem saberia da mudança.
