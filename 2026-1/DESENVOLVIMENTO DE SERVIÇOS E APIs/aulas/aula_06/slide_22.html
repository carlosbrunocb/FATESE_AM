Antes vs Depois dos Middlewares

Comparação do arquivo livroController.js

Sem Middleware Acoplado
1const criarLivro = (req, res) => {
2 const { titulo, autor } = req.body;
3
4 // ⚠️ Validação mistura com regra de negócio
5 if (!titulo || !autor) {
6 return res.status(400).json({
7 erro: "Dados Inválidos"
8 });
9 }
10
11 // Lógica de criação...
12 db.save({ titulo, autor });
13 res.status(201).send();
14};

Problemas

O Controller viola o princípio da responsabilidade única. Se a regra de validação mudar, você precisa editar o Controller. O código fica "sujo" e repetitivo em cada método.

Com Middleware Limpo
1const criarLivro = (req, res) => {
2 // ✅ Validação já ocorreu antes!
3 // O req.body chega aqui 100% confiável
4 const { titulo, autor } = req.body;
5
6 // Foco total na Regra de Negócio
7 db.save({ titulo, autor });
8
9 res.status(201).send();
10};
11
12// O erro, se houver, nem chega aqui.
13// Foi barrado pelo middleware anterior.

Benefícios

O Controller fica focado apenas em "O que fazer com os dados". A lógica de "Se os dados estão certos" é delegada, tornando o código mais legível e fácil de testar.
