O que é uma API RESTful? (Revisão)

REST (REpresentational State Transfer) é um estilo de arquitetura que define como serviços web devem se comunicar. Ele usa os verbos HTTP padrões que já conhecemos:

  • GET: Para ler dados.
  • POST: Para criar um novo dado.
  • PUT / PATCH: Para atualizar um dado existente.
  • DELETE: Para remover um dado.

Nossa API é "RESTful" porque mapeia esses verbos a ações claras sobre o recurso /users: 

// Em user.routes.js
router.get('/users', userController.getAll);        // Lê todos
router.get('/users/:id', userController.getById);      // Lê um
router.post('/users', userController.create);       // Cria um

A Arquitetura: Separando Responsabilidades

Não colocamos todo o código em um arquivo só. Nós separamos a API em camadas, cada uma com uma única responsabilidade. Isso é essencial para manutenção e testabilidade.

1. Camada de Roteamento

Arquivo: user.routes.js

Define as URLs e os verbos HTTP. É o "GPS" da aplicação.

2. Camada de Controle

Arquivo: user.controller.js

Recebe a requisição (req) e envia a resposta (res). É o "Gerente" que delega tarefas.

3. Camada de Serviço

Arquivo: user.service.js

Contém a Lógica de Negócio. É o "Cérebro" que executa as tarefas.

Fluxo: Rota ➡️ Controlador ➡️ Serviço ➡️ Controlador ➡️ Resposta

1. Camada de Roteamento (O "GPS")

Arquivo: src/routes/user.routes.js

A responsabilidade desta camada é simples: ela diz ao Express "Quando uma requisição X chegar nesta URL, execute a função Y".

Note que ela não sabe como criar um usuário, apenas quem chamar (o userController).

const { Router } = require('express');
const userController = require('../controllers/user.controller');
const { validateUserCreation } = require('../middlewares/validation.middleware');

const router = Router();

// Rota para listar todos os usuários
router.get('/users', userController.getAll);

// Rota para criar um usuário
// Note a cadeia: primeiro valida, DEPOIS cria
router.post(
    '/users',
    validateUserCreation, // <- Middleware de Validação
    userController.create   // <- Controlador
);

module.exports = router;

2. Camada de Controle (O "Gerente")

Arquivo: src/controllers/user.controller.js

O Controlador é o "gerente" da requisição. Ele é o único que fala com req (requisição) e res (resposta). Ele recebe o pedido, chama o Serviço para fazer o trabalho pesado, e depois envia a resposta ao cliente.

const userService = require('../services/user.service');

class UserController {
    
    async create(req, res, next) {
        try {
            // 1. Pega os dados do corpo da requisição
            const userData = req.body;

            // 2. DELEGA o trabalho para o Serviço
            const newUser = userService.create(userData);

            // 3. ENVIA a resposta de sucesso
            res.status(201).json({ success: true, data: newUser });

        } catch (error) {
            // 4. Se o Serviço falhar, DELEGA o erro
            next(error);
        }
    }
}

3. Camada de Serviço (O "Cérebro")

Arquivo: src/services/user.service.js

Esta é a camada mais importante: a Lógica de Negócio. O Serviço executa a tarefa. Ele não sabe o que é req ou res. Ele apenas recebe dados (userData), faz seu trabalho (salva no banco, cria ID) e retorna um resultado ou lança um erro.

const db = require('../utils/database.util'); // Nosso "Banco"
const AppError = require('../errors/AppError');
const DB_KEY = 'users';

class UserService {
    
    create(userData) {
        // REGRA DE NEGÓCIO 1: Validar email
        if (!userData.email) {
            throw new AppError('O campo e-mail é obrigatório.', 400);
        }

        const users = db.getData(DB_KEY);

        // REGRA DE NEGÓCIO 2: Gerar um ID
        const lastId = users.length > 0 ? users[users.length - 1].id : 0;
        const newUser = { id: lastId + 1, ...userData };

        users.push(newUser);
        db.saveData(DB_KEY, users); // Ação de persistência
        
        return newUser;
    }
}

O que é Middleware? (Os "Porteiros")

Middleware são funções que rodam no meio do ciclo da requisição (Requisição ➡️ ...Middleware... ➡️ Resposta).

Pense neles como "porteiros" ou "checkpoints" que uma requisição deve passar.

A Anatomia de um Middleware

Ele sempre recebe 3 argumentos (ou 4, se for de erro):

function meuMiddleware(req, res, next) {
    // ... faz alguma coisa (ex: logar, checar algo) ...

    // REGRA DE OURO:
    // Ou você BARRA a requisição...
    // return res.status(401).json({ message: 'Não autorizado' });

    // ...Ou você DEIXA PASSAR para o próximo.
    next();
}

Se você não enviar uma resposta e nem chamar next(), a requisição fica "travada" (timeout).

Middleware em Ação (Nosso Projeto)

Nós usamos 2 tipos principais de middleware:

1. Middleware Global (Logger)

Arquivo: logger.middleware.js

É registrado no server.js com app.use(logger). Ele roda em todas as requisições que chegam na API, sem exceção.

function logger(req, res, next) {
  console.log(
    `[${new Date().toISOString()}] ${req.method} ${req.url}`
  );
  next(); // Deixa passar
}

2. Middleware Específico (Validação)

Arquivo: validation.middleware.js

É registrado no user.routes.js apenas para a rota POST /users. Ele atua como um "segurança" que barra dados inválidos.

function validateUserCreation(req, res, next) {
  const { name, email } = req.body;

  if (!name || !email) {
    // BARRA a requisição
    return res.status(400).json({ ... });
  }
  
  next(); // Deixa passar
}

Tratamento de Erros (A "Rede de Segurança")

O que acontece quando o userService lança um erro? O servidor quebra? Não! Graças à nossa arquitetura.

1. O Erro Customizado: AppError.js

Criamos uma classe de Erro que nos permite definir uma mensagem E um statusCode. Isso é crucial para o REST (ex: 404 Not Found, 400 Bad Request).

// Em user.service.js
throw new AppError('Usuário não encontrado.', 404);

2. O Middleware de Erro: errorHandler.middleware.js

Este é um middleware especial com 4 argumentos (err, req, res, next). Ele é registrado por último no server.js.

function errorHandler(err, req, res, next) {
  // Pega o statusCode do AppError, ou 500 (Erro Genérico)
  const statusCode = err.statusCode || 500;
  
  res.status(statusCode).json({
      success: false,
      message: err.message
  });
}

O try...catch(next) no Controlador é o que joga o erro do Serviço para este "hospital" centralizado.

Fluxo Completo: POST /api/users (Sucesso)

Vamos seguir uma requisição pela nossa API:

  • 1. Requisição Cliente envia POST /api/users com { "name": "Ana", "email": "ana@senac.com" }.
  • 2. Middlewares Globais (em server.js) express.json() lê o corpo. logger() imprime no console e chama next().
  • 3. Rota (em user.routes.js) Express encontra router.post('/users', ...).
  • 4. Middleware Específico (em validation.middleware.js) validateUserCreation checa: name existe? Sim. email existe? Sim. Chama next().
  • 5. Controlador (em user.controller.js) userController.create é chamado. Entra no try.
  • 6. Serviço (em user.service.js) userService.create(...) é chamado. Valida o email (ok), lê o DB, cria o ID (ex: 1), salva no DB. Retorna { id: 1, ... }.
  • 7. Resposta (de volta ao user.controller.js) res.status(201).json(...) é executado.
  • 8. FIM Cliente recebe a resposta 201 Created com os dados do usuário.

Fluxo Completo: POST /api/users (Erro de Validação)

Agora, o que acontece se o cliente errar?

  • 1. Requisição Cliente envia POST /api/users com { "name": "Ana" } (esqueceu o email!).
  • 2. Middlewares Globais (em server.js) express.json() e logger() rodam normalmente e chamam next().
  • 3. Rota (em user.routes.js) Express encontra router.post('/users', ...).
  • 4. Middleware Específico (em validation.middleware.js) validateUserCreation checa: name existe? Sim. email existe? Não.
  • 5. Resposta (Erro) O middleware BARRA a requisição. Ele executa return res.status(400).json(...).
  • 6. FIM O userController nunca é executado. O cliente recebe 400 Bad Request.

Fluxo Completo: GET /api/users/99 (Erro de Serviço)

E se o erro acontecer na lógica de negócio?

  • 1. Requisição Cliente envia GET /api/users/99.
  • 2. Rota e Controlador logger() roda. user.routes.js chama userController.getById.
  • 3. Serviço (em user.service.js) userService.getById(99) é chamado. Ele procura no DB e não encontra.
  • 4. O Erro é Lançado (throw) O Serviço executa: throw new AppError('Usuário não encontrado.', 404).
  • 5. O Controlador Pega (catch) O try...catch do userController captura o erro.
  • 6. Delegação (next) O Controlador executa next(error).
  • 7. O "Hospital" (em server.js) O Express pula tudo e entrega o error para o errorHandler(err, req, res, next).
  • 8. FIM O errorHandler envia a resposta: 404 Not Found.
O Fluxo Completo (Erro de Serviço)
