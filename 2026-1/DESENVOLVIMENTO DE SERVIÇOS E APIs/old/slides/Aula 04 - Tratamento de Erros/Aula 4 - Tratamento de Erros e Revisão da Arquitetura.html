O Problema: try...catch em todo controller

Repetição de código e ruído visual

Blocos try-catch idênticos em cada método do controller, aumentando o tamanho do código e dificultando a leitura.

Inconsistência nas respostas de erro

Diferentes desenvolvedores podem implementar formatos de resposta distintos, prejudicando a consistência da API.

Dificulta observabilidade e manutenção

Logs e tratamentos espalhados tornam difícil rastrear, monitorar e corrigir erros de forma centralizada.

Aumenta chance de esquecer tratamentos

Desenvolvedores podem esquecer de adicionar try-catch em novos métodos, resultando em falhas não tratadas.

Anti-DRY (Don't Repeat Yourself)

Viola o princípio fundamental de não repetir código, dificultando mudanças e evoluções na forma de tratar erros.

A solução: middleware de erro centralizado

Um único ponto para tratar e formatar erros

Centraliza toda a lógica de tratamento de erros em um lugar, facilitando manutenção e garantindo consistência.

Remove repetição nos controllers (DRY)

Controllers ficam mais limpos e focados apenas na lógica de negócio, delegando erros ao middleware especializado.

Facilita logging e observabilidade

Todos os erros são registrados em um único ponto, melhorando diagnósticos e permitindo implementação de telemetria.

Permite padronizar payloads de erro

Garante formato consistente nas respostas de erro, melhorando experiência para consumidores da API.

Simplifica evolução futura

Facilita implementação de recursos avançados como internacionalização (i18n), IDs de correlação ou integração com ferramentas como Sentry.

A Solução: O Princípio (Separação de Preocupações)

O princípio da Separação de Preocupações (Separation of Concerns) nos diz:

  • O Controller deve se preocupar em orquestrar o fluxo (receber a requisição, chamar o serviço, enviar a resposta).
  • O Serviço deve se preocupar com a lógica de negócio.
  • Um Middleware de Erro deve se preocupar em capturar, formatar e enviar todas as respostas de erro.

Vamos criar um "para-raios" global para todos os erros da nossa aplicação.

Usaremos um tipo especial de middleware do Express para isso.

Benefícios da centralização

Consistência e previsibilidade das respostas

Todos os erros são formatados da mesma maneira, criando uma API com comportamento consistente e previsível para os clientes.

Menor acoplamento nos controllers

Controllers ficam mais enxutos e focados apenas na lógica de coordenação, delegando o tratamento de erro para o middleware.

Logs mais úteis e concentrados

Centraliza logs de erro em um único ponto, facilitando o monitoramento e a identificação de problemas em produção.

Facilidade para métricas e alertas

Possibilita registrar estatísticas de erro e criar alertas para monitoramento a partir de um único ponto de observação.

Melhora DX e manutenção

Developer Experience (DX) aprimorada com menos código repetitivo e maior facilidade para evoluir o tratamento de erros globalmente.

Anatomia do Middleware de Erro

Função com 4 parâmetros

É definida como function(err, req, res, next) - O Express reconhece como middleware de erro pela quantidade exata de parâmetros.

Posição no pipeline

Deve ser registrado após todas as rotas com app.use(errorHandler) para capturar erros ocorridos em qualquer rota ou middleware anterior.

Formatação da resposta

Define o status code (usando err.statusCode || 500) e o corpo da resposta em formato padronizado e consistente.

Diferenciação por ambiente

Pode comportar-se de forma diferente em desenvolvimento vs. produção (ex.: expor stack trace apenas em ambiente de dev).

Enriquecimento com metadados

Pode adicionar informações úteis como timestamp, path, requestId ou correlationId para facilitar troubleshooting e observabilidade.

Os 4 argumentos: (err, req, res, next)

err

Instância do erro lançado ou passado para next(). Contém a mensagem de erro (err.message), stack trace (err.stack) e pode conter propriedades customizadas como statusCode ou errors.

req

Objeto de requisição HTTP com informações do contexto: método, URL, parâmetros (req.params), corpo (req.body) e cabeçalhos. Útil para incluir detalhes da requisição nos logs de erro.

res

Objeto de resposta HTTP usado para enviar dados ao cliente. No middleware de erro, utilizamos res.status(code).json(payload) para formatar a resposta de erro padronizada.

next

Função que encadeia para o próximo middleware. Raramente utilizada no handler de erro, exceto para delegar a outro middleware específico (ex: divisão entre erros operacionais e de programação).

Anatomia: Comparação da Assinatura

O Express identifica um middleware de tratamento de erro de forma única: pela sua assinatura (seus argumentos).

Middleware Comum (3 argumentos)

Usado para logs, validação, autenticação, etc.

function logger(req, res, next) {
  console.log(req.method, req.path);
  next(); // Passa para o próximo
}

Middleware de Erro (4 argumentos)

Executado apenas quando `next(error)` é chamado.

function errorHandler(err, req, res, next) {
  // O 'err' é o primeiro argumento!
  console.error(err);
  res.status(500).json(/* ... */);
}

Padronização das respostas de erro

Estrutura recomendada

{ success: false, message: "...", errors?: [...] }

Mensagens claras para clientes

Respostas devem ser compreensíveis para humanos, direcionando o cliente sobre como resolver o problema.

Campos opcionais para validação

Use o array errors para listar múltiplas falhas de validação, com campo, valor e mensagem específica.

Segurança em produção

Não vazar detalhes sensíveis como mensagens de erro do banco de dados, stack traces ou caminhos do sistema.

Rastreabilidade com traceId

Incluir identificadores de correlação (opcional) que permitem associar logs de servidor com a requisição específica.

Status Codes HTTP comuns

400

Bad Request

Erros de validação ou formato de entrada inválido. Cliente enviou uma requisição que o servidor não consegue processar.

401

Unauthorized

Autenticação necessária. Cliente precisa se identificar antes de acessar o recurso (token ausente ou inválido).

403

Forbidden

Cliente autenticado, mas sem permissão. O servidor entende a requisição, mas recusa a autorização.

404

Not Found

Recurso inexistente. O endpoint existe, mas o recurso específico (por ID, por exemplo) não foi encontrado.

409

Conflict

Conflito de regra de negócio. Ex: tentativa de cadastrar email já existente ou violar restrição única.

500

Internal Server Error

Erro interno/inesperado do servidor. Problema que não foi antecipado ou tratado corretamente pelo backend.

Como o Express sabe qual chamar?

O Express mantém internamente duas "pilhas" de middlewares: a pilha principal e a pilha de erros.

  • Quando você chama `next()` (sem argumento):

    O Express procura o próximo middleware na pilha principal (de 3 argumentos).

  • Quando você chama `next(error)` (com argumento):

    O Express ignora todos os middlewares restantes da pilha principal e procura o primeiro middleware na pilha de erros (de 4 argumentos).

Importante: Se você não chamar `next()` nem enviar uma resposta (`res.send()`, `res.json()`, etc.), a requisição ficará "presa" (pending) e dará timeout.

Passo 1: Criando o Error Handler

Criar arquivo especializado

Crie o arquivo src/middlewares/errorHandler.middleware.js para centralizar toda lógica de tratamento de erros.

Garantir os 4 parâmetros

A função deve receber exatamente 4 parâmetros: (err, req, res, next) para o Express reconhecê-la como middleware de erro.

Definir status code e mensagem

Extrair statusCode do erro ou usar 500 como fallback; garantir sempre uma mensagem amigável para o cliente.

Exportar middleware

Exportar a função com module.exports = errorHandler para reutilização na aplicação.

Mão na Massa (Passo 1): O Código do Handler

Vamos criar nosso tratador centralizado. Ele será responsável por formatar todas as respostas de erro da API.

Arquivo: `src/middlewares/errorHandler.middleware.js`

// Este middleware DEVE ter 4 argumentos para ser reconhecido
// como um middleware de erro pelo Express.
function errorHandler(err, req, res, next) {

  // 1. Loga o erro no console (essencial para debug)
  console.error('ERRO DETECTADO:', err.message);
  console.error(err.stack); // Opcional, mas útil

  // 2. Define um status code padrão (500)
  const statusCode = err.statusCode || 500;

  // 3. Envia uma resposta JSON padronizada
  res.status(statusCode).json({
    success: false,
    message: err.message || 'Ocorreu um erro interno no servidor.'
  });
}

module.exports = errorHandler;

Passo 2: Registrando o Middleware

Importar errorHandler no index.js

Importe o middleware de erro criado usando require para disponibilizá-lo na aplicação principal.

Registrar após as rotas com app.use(errorHandler)

Adicione o handler após a definição de todas as rotas da aplicação para capturar erros em qualquer endpoint.

Ordem dos middlewares é crucial

Garanta que middlewares de parsing (express.json()) e logging venham antes das rotas, e o errorHandler após todas as rotas.

O handler DEVE ser o último middleware

Para capturar erros de qualquer middleware anterior, inclusive das próprias rotas, o errorHandler precisa ser o último app.use() na aplicação.

A ordem importa!

0

app = express()

1

express.json()

2

logger

3

/api (rotas)

4

errorHandler

ÚLTIMO!

Pipeline recomendado:

1) app.use(express.json()) - Parse do corpo da requisição

2) app.use(logger) - Logging das requisições

3) app.use('/api', rotas) - Rotas da API

4) app.use(errorHandler) - Tratamento de erros

Por quê? O Express encontra o primeiro handler de erro após o next(err). Se o errorHandler vier antes das rotas, os erros gerados nas rotas não serão capturados.

Mão na Massa (Passo 2): Registrando no index.js

A ordem de registro dos middlewares no `index.js` (ou `app.js`) é CRUCIAL.

O middleware de tratamento de erros DEVE SER O ÚLTIMO `app.use()` registrado. 

// index.js
const express = require('express');
const logger = require('./middlewares/logger.middleware');
const errorHandler = require('./middlewares/errorHandler.middleware');
const userRoutes = require('./routes/user.routes');

const app = express();

// --- 1. Middlewares Globais ---
app.use(express.json()); 
app.use(logger);         

// --- 2. Rotas da Aplicação ---
app.use('/api', userRoutes);
// ... outras rotas ...

// --- 3. Middlewares de Erro (SEMPRE POR ÚLTIMO!) ---
app.use(errorHandler);

app.listen(3000, () => console.log('Servidor rodando...'));

Passo 3: Refatorando os Controllers

Envolva a lógica em try/catch

Encapsule todo o código do controller em um bloco try e capture exceções no bloco catch, evitando erros não tratados.

Encaminhe erros com next(error)

No bloco catch, use next(error) para delegar o tratamento ao middleware de erro centralizado, em vez de tratar no próprio controller.

Evite responder duas vezes

Nunca use res.json() após chamar next(error). Quando next(error) é chamado, o middleware de erro assume a responsabilidade pela resposta.

Mantenha controllers enxutos

Controllers devem apenas orquestrar chamadas ao serviço e responder ao cliente. Deixe regras de negócio e validações complexas nos serviços.

Facilite os testes unitários

Controllers mais simples e com responsabilidades claras são mais fáceis de testar isoladamente, melhorando a cobertura de testes.

Mão na Massa (Passo 3): Refatorando o Controller

Agora, vamos modificar o `user.controller.js` para delegar o erro para o nosso `errorHandler`. 

// src/controllers/user.controller.js
const userService = require('../services/user.service');

class UserController {
  
  async create(req, res, next) { // 1. Adicionamos o 'next'
    try {
      // 2. Lógica de sucesso
      const newUser = userService.create(req.body);
      res.status(201).json({ success: true, data: newUser });
    
    } catch (error) {
      // 3. Se der erro, DELEGUE!
      next(error); 
    }
  }

  // ... (refatorar todos os outros métodos) ...
}

module.exports = new UserController();

Atenção: `async/await` e Promises

No Express (versão 4 e anteriores), se um erro acontece dentro de uma função `async` (uma *Promise Rejeitada*), o Express não o captura automaticamente.

É por isso que o `try...catch` é obrigatório ao usar `async/await` nos controllers. 

// ISSO QUEBRA O SERVIDOR! (Sem try...catch)
app.get('/erro-async', async (req, res, next) => {
  // Se db.find() falhar, o Express não captura.
  const data = await db.find(); 
  res.json(data);
});

// ISSO FUNCIONA! (Com try...catch)
app.get('/erro-async-correto', async (req, res, next) => {
  try {
    const data = await db.find(); 
    res.json(data);
  } catch (error) {
    next(error); // O erro é pego e enviado ao errorHandler
  }
});

(O Express 5 introduz melhorias nisso, mas o padrão `try...catch` ainda é a prática mais segura).

Aprofundando: Classes de Erro Customizadas

E se quisermos lançar erros com status codes específicos? (Ex: 404 Not Found, 400 Bad Request).

Lançar um `new Error()` sempre resultará em um status 500 (porque `err.statusCode` será `undefined`).

Solução: Criar nossa própria classe `AppError` que armazena o `statusCode`. 

// src/errors/AppError.js
class AppError extends Error {
  constructor(message, statusCode) {
    super(message); // Passa a mensagem para a classe Error (pai)
    
    this.statusCode = statusCode;
    this.name = this.constructor.name; // Garante o nome correto

    Error.captureStackTrace(this, this.constructor);
  }
}

module.exports = AppError;

Usando o `AppError` no Serviço

Agora, em vez de lançar um `new Error()`, podemos lançar um `new AppError()` com a mensagem e o status corretos. 

// src/services/user.service.js
const AppError = require('../errors/AppError');
const usersDB = []; // Nosso "banco" em memória

class UserService {
  
  create(userData) {
    if (!userData.email) {
      // Lança um erro 400 (Bad Request)
      throw new AppError('O campo e-mail é obrigatório.', 400);
    }
    // ... lógica ...
    return newUser;
  }

  getById(id) {
    const user = usersDB.find(u => u.id === id);
    if (!user) {
      // Lança um erro 404 (Not Found)
      throw new AppError('Usuário não encontrado.', 404);
    }
    return user;
  }
}

Nosso `errorHandler` já está pronto para isso! Ele usará o `err.statusCode` (400 ou 404) em vez do padrão 500.

Bônus: Automatizando o `try...catch`

Ainda estamos repetindo `try...catch(next)` em todo método de controller. Podemos abstrair isso também!

Podemos criar uma função "wrapper" (embrulho) que adiciona o `catch` para nós. 

// src/utils/catchAsync.js
// Esta função recebe uma função de controller (fn)
const catchAsync = (fn) => {
  // E retorna uma nova função que o Express irá chamar
  return (req, res, next) => {
    // Executamos a função original (fn)
    // Se ela for uma Promise e falhar (rejeitar),
    // o .catch(next) pegará o erro e o enviará para o errorHandler.
    fn(req, res, next).catch(next);
  };
};

module.exports = catchAsync;

Bônus: Usando o `catchAsync`

Agora, nosso controller fica incrivelmente limpo. Não precisamos mais de `try...catch` nele. 

// src/controllers/user.controller.js
const userService = require('../services/user.service');
const catchAsync = require('../utils/catchAsync'); // Importamos

class UserController {
  
  // Embrulhamos nosso método async com o catchAsync
  create = catchAsync(async (req, res, next) => {
    // Sem try...catch aqui!
    const newUser = userService.create(req.body);
    res.status(201).json({ success: true, data: newUser });
  });

  getById = catchAsync(async (req, res, next) => {
    const user = userService.getById(req.params.id);
    res.status(200).json({ success: true, data: user });
  });

  // ...
}

module.exports = new UserController();
