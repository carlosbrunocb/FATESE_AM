🤔 O Problema: O "Monstro" index.js

Na nossa primeira aula, criamos um servidor e uma rota "Hello World", tudo no mesmo arquivo index.js.

Isso funciona para um exemplo simples, mas...

O que acontece quando tivermos 50 rotas? (Usuários, Produtos, Pedidos, Pagamentos...)

  • O arquivo index.js se tornará gigante e ilegível.
  • A lógica de negócio (como calcular um frete) ficará misturada com a lógica de HTTP (req, res).
  • Dar manutenção será um pesadelo.
  • Testar partes específicas do código será quase impossível.

Precisamos de organização!

💡 A Solução: Arquitetura em Camadas

Vamos organizar nosso código dividindo as responsabilidades em "camadas" lógicas.

O princípio por trás disso é chamado de:

Separação de Responsabilidades
(Separation of Concerns - SoC)

A ideia é simples: cada parte do seu código deve ter uma, e apenas uma, responsabilidade.

  • Um arquivo que define rotas SÓ deve definir rotas.
  • Um arquivo que tem lógica de negócio SÓ deve ter lógica de negócio.

Camada 1: Rotas (Routes)

É o "Porteiro" da sua aplicação.

  • Responsabilidade Única: Definir os endpoints (URLs) e quais verbos HTTP (GET, POST, PUT, DELETE) eles aceitam.
  • Sabe qual rota (ex: /users) deve chamar qual função do Controller (ex: userController.getAll).
  • Não sabe NADA sobre lógica de negócio.
  • Não sabe NADA sobre banco de dados.
  • Apenas direciona o tráfego.

Exemplo: "Ah, uma requisição GET para /users/:id? Mande para o userController.getById."

Camada 2: Controladores (Controllers)

É o "Orquestrador" da requisição.

  • Responsabilidade Única: Orquestrar o fluxo da requisição.
  • Recebe os objetos req (Requisição) e res (Resposta).
  • Extrai dados da requisição (ex: req.body, req.params.id).
  • Chama a camada de Serviço para executar a lógica de negócio, passando os dados necessários.
  • Recebe a resposta do Serviço e envia de volta ao cliente (ex: res.json(...) ou res.status(404).json(...)).
  • Não deve conter lógica de negócio! (Ex: NUNCA fazer um cálculo de desconto aqui).

Camada 3: Serviços (Services)

É o "Cérebro" da aplicação. Aqui mora a lógica de negócio.

  • Responsabilidade Única: Conter a lógica de negócio pura (Business Logic).
  • Define "COMO" as coisas são feitas.
    • "Como buscar um usuário no banco de dados."
    • "Como validar se um CPF é válido."
    • "Como calcular o total de um pedido."
  • Não deve saber sobre req ou res!
  • Recebe dados puros (ex: um id ou um objeto userData) e retorna dados puros (ex: um objeto user ou um array).
  • Isso torna a lógica de negócio testável e reutilizável.

🧑‍💻 Mão na Massa: Estruturando o Projeto

Vamos organizar nosso projeto! Crie esta estrutura de pastas:

/ (pasta_do_projeto)
├── node_modules/
├── src/
│   ├── controllers/
│   │   └── user.controller.js
│   ├── routes/
│   │   ├── index.js         (Opcional, para juntar rotas)
│   │   └── user.routes.js
│   ├── services/
│   │   └── user.service.js
│   └── index.js         (Nosso arquivo principal)
├── package.json
└── package-lock.json

(Mova seu index.js para dentro da pasta /src)

⚙️ Facilitando a Vida: Nodemon

Cansado de ter que parar e iniciar o servidor (node src/index.js) a cada mudança?

O Nodemon reinicia automaticamente o servidor quando detecta alterações nos arquivos.

1. Instale como uma dependência de desenvolvimento (-D):

npm install nodemon -D

2. Configure seu package.json para usá-lo:

// package.json
{
  ...
  "scripts": {
    "start": "node src/index.js",
    "dev": "nodemon src/index.js"
  },
  ...
}

Agora, rode npm run dev para iniciar o servidor!

🧱 O Middleware express.json()

Quando um cliente envia dados para nossa API (ex: em uma requisição POST), ele geralmente envia em formato JSON no "corpo" (body) da requisição.

Por padrão, o Express não sabe como ler esse JSON.

Precisamos adicionar um middleware (um "plugin") nativo do Express para "traduzir" esse JSON para um objeto JavaScript que possamos usar em req.body.

Adicione esta linha ao seu src/index.js, antes de definir as rotas:

// Em src/index.js
const express = require('express');
const app = express();

// Middleware para ler JSON
app.use(express.json());

// ... resto do seu código (rotas, etc) ...

Refatoração: Camada de Serviço

src/services/user.service.js

Aqui fica a lógica de negócio. Por enquanto, vamos simular um banco de dados com um array.

// Por enquanto, simulando um banco de dados
const users = [{ id: 1, name: 'João' }];

class UserService {
  getAll() {
    // Lógica de negócio (ex: buscar no DB)
    return users;
  }

  create(userData) {
    // Lógica de negócio (ex: validar dados, inserir no DB)
    const newUser = { id: users.length + 1, ...userData };
    users.push(newUser);
    return newUser;
  }

  getById(id) {
     // Lógica (ex: buscar por ID)
     return users.find(u => u.id === parseInt(id));
  }
}

// Exportamos uma *instância* da classe
module.exports = new UserService();

Refatoração: Camada de Controle

src/controllers/user.controller.js

O orquestrador. Importa o Serviço, chama os métodos e envia a resposta.

const userService = require('../services/user.service');

class UserController {
  // Método para buscar todos os usuários
  async getAll(req, res) {
    const users = userService.getAll();
    res.json({ success: true, data: users });
  }

  // Método para criar um novo usuário
  async create(req, res) {
    // req.body - Vem do middleware express.json()
    const newUser = userService.create(req.body);
    // 201 = Created (Sucesso ao criar)
    res.status(201).json({ success: true, data: newUser });
  }
  
  // ... continua no próximo slide ...
}

Refatoração: Camada de Controle (Cont.)

src/controllers/user.controller.js

Continuando nosso Controller, agora com o método getById.

// ... continuação ...

class UserController {
  // ... métodos getAll e create ...

  // Método para buscar usuário por ID
  async getById(req, res) {
     // req.params - Vem da URL (ex: /users/1)
     const user = userService.getById(req.params.id);
     
     if (!user) {
        // 404 = Not Found
        return res.status(404).json({ success: false, message: 'Usuário não encontrado' });
     }
     
     res.json({ success: true, data: user });
  }
}

module.exports = new UserController();

🎁 Parâmetros de Requisição

De onde vêm os dados que o Controller recebe? (req)

  • req.params (Route Params)
    • Vêm como parte da URL, para identificar um recurso específico.
    • Ex: /users/:id (/users/1) ➡️ req.params.id será "1".
  • req.body (Request Body)
    • Vêm no "corpo" da requisição, invisível na URL.
    • Usado para enviar dados complexos (JSON).
    • Ex: POST /users com JSON {"name": "Maria"} ➡️ req.body.name será "Maria".
    • (Requer app.use(express.json()))
  • req.query (Query Params)
    • Vêm no final da URL após ?, para filtros, busca ou paginação.
    • Ex: /users?search=joao&page=2 ➡️ req.query.search será "joao".

Refatoração: Camada de Rotas

src/routes/user.routes.js

O "Porteiro". Define os caminhos e os conecta aos métodos do Controller.

const { Router } = require('express');
const userController = require('../controllers/user.controller');

const router = Router();

// Rota para buscar todos os usuários
router.get('/users', userController.getAll);

// Rota para criar um novo usuário
router.post('/users', userController.create);

// Rota com parâmetro (req.params.id)
router.get('/users/:id', userController.getById);

module.exports = router;

Refatoração: Arquivo Principal

src/index.js

Nosso index.js ficou muito mais limpo! Ele só configura o servidor e "usa" as rotas.

const express = require('express');
// Importamos o arquivo de rotas de usuário
const userRoutes = require('./routes/user.routes');

const app = express();
const PORT = 3000;

// Middleware para ler JSON
app.use(express.json());

// Usar as rotas de usuário
// Todas as rotas em 'userRoutes' terão o prefixo '/api'
app.use('/api', userRoutes); // Ex: /api/users, /api/users/1

app.listen(PORT, () => {
  console.log(`Servidor rodando em http://localhost:${PORT}`);
});

🌀 O Fluxo Completo (Revisão)

O que acontece em uma requisição POST /api/users com o JSON {"name": "Maria"}?

  1. O Insomnia envia a requisição.
  2. O index.js (servidor) recebe.
  3. O middleware express.json() lê o body {"name": "Maria"} e o coloca em req.body.
  4. O app.use('/api', userRoutes) vê o prefixo /api e passa a requisição para user.routes.js.
  5. O router.post('/users', ...) bate com a rota e o verbo. Ele chama userController.create.
  6. O userController.create é executado. Ele pega req.body.
  7. Ele chama userService.create(req.body), passando {"name": "Maria"}.
  8. O userService.create (lógica de negócio) cria o newUser (ex: {id: 2, name: "Maria"}) e o retorna.
  9. O userController recebe newUser e envia a resposta: res.status(201).json(...).
  10. O Insomnia recebe a resposta JSON com status 201.

🧪 Hora de Testar! (Insomnia)

Abra o Insomnia (ou Postman) e teste todas as rotas que criamos (verifique se seu servidor está rodando com npm run dev):

  1. Criar um usuário (POST)

  2. Buscar todos os usuários (GET)
    • Verbo: GET
    • URL: http://localhost:3000/api/users
    • Clique "Send". Você deve ver a lista com "João" e "Maria".
  3. Buscar por ID (GET)
    • Verbo: GET
    • URL: http://localhost:3000/api/users/2
    • Clique "Send". Você deve ver apenas "Maria".
