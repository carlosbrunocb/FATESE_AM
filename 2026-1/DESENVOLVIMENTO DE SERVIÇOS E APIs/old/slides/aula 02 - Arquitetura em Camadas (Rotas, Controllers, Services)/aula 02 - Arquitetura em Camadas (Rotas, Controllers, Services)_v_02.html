1. Teoria: Separação de Responsabilidades

"Separation of Concerns" (SoC)

É um princípio de design de software que diz:

"Cada parte do software deve ter uma única e bem definida responsabilidade."

Em vez de um arquivo que faz TUDO (define rotas, valida dados, acessa o banco, formata resposta), teremos arquivos ESPECIALISTAS.

2. Teoria: A Arquitetura em 3 Camadas

Vamos dividir nosso projeto em 3 camadas principais:

Requisição HTTP ➔ Rota ➔ Controller ➔ Service ➔ Banco de Dados (e o caminho reverso para a Resposta)

A Camada de ROTA (Router)

O "Recepcionista" 🤵

Responsabilidade Única:

Definir os endpoints (URLs) e quais verbos HTTP (GET, POST, etc.) eles aceitam.

Ele NÃO SABE o que fazer. Ele apenas DIRECIONA a requisição para o especialista (o Controller).

Ex: "Uma requisição GET para /users? Fale com o userController."

A Camada de CONTROLE (Controller)

O "Gerente da Loja" 👨‍💼

Responsabilidade Única:

Orquestrar o fluxo da requisição. Ele é o intermediário entre a Rota e o Serviço.

1. Recebe a requisição (req, res) da Rota.

2. Extrai dados (req.body, req.params).

3. Chama o Serviço (a lógica de negócio) para fazer o trabalho.

4. Pega a resposta do Serviço e envia de volta ao cliente (res.json(), res.status()).

A Camada de SERVIÇO (Service)

O "Especialista" / "Estoque" 👩‍🔧

Responsabilidade Única:

Conter a LÓGICA DE NEGÓCIO pura da aplicação.

Aqui fica: Acesso ao banco de dados, cálculos complexos, validações de regra de negócio (ex: "o usuário é maior de idade?").

CRÍTICO: O Serviço NUNCA deve saber sobre req ou res. Ele recebe dados puros (ex: `id`, `userData`) e retorna dados puros (ex: um `user`).

4. Ferramenta: nodemon

Chega de Ctrl+C, node index.js!

O que é? O nodemon (Node Monitor) é uma ferramenta que "assiste" seus arquivos.

O que faz? Assim que você salva uma alteração em um arquivo .js, ele automaticamente reinicia o servidor para você.

Por que? Aumenta drasticamente a produtividade. Você salva o código e testa no Insomnia imediatamente.

Instalando o nodemon

Vamos instalá-lo como uma Dependência de Desenvolvimento:

npm install nodemon -D

dependencies vs devDependencies (-D)

  • dependencies (Ex: express):

    Pacotes que a aplicação precisa para rodar em produção.

  • devDependencies (Ex: nodemon):

    Pacotes usados APENAS durante o desenvolvimento (testes, monitoramento, etc.).

Configurando o package.json

Vamos criar um "atalho" (script) no package.json para rodar o nodemon: 

// package.json
...
"scripts": {
  "test": "echo \"Error: no test specified\" && exit 1",
  "dev": "nodemon src/index.js"
},
...

Agora, em vez de node src/index.js, vamos rodar no terminal: 

npm run dev

Seu servidor irá iniciar e ficará "assistindo" por mudanças.

5. O Middleware express.json()

Na Aula 1, nossa API só enviava dados (res.send()).

Mas como fazemos para RECEBER dados do cliente? (Ex: em uma requisição POST para criar um usuário).

Por padrão, o Express não sabe como "ler" o JSON enviado no corpo (body) da requisição.

Precisamos ativar um "tradutor" (Middleware) nativo do Express: 

app.use( express.json() );

O que é o req.body?

Quando o express.json() está ativo, qualquer JSON enviado pelo cliente (via Insomnia, por ex.) será "traduzido" e colocado dentro do objeto req.body.

Requisição do Insomnia (Body):

{
  "name": "Bruno",
  "email": "bruno@senac.com"
}

No nosso Controller (Código):

console.log(req.body);
// Output:
// { name: 'Bruno', email: 'bruno@senac.com' }

const nome = req.body.name; // "Bruno"

6. Plano de Refatoração

Vamos migrar nosso código do index.js antigo para a nova estrutura.

A melhor forma é de "dentro para fora":

  • Passo 1. Criar o Serviço (A lógica pura).
  • Passo 2. Criar o Controlador (O orquestrador).
  • Passo 3. Criar a Rota (O endpoint).
  • Passo 4. Limpar o index.js principal.

Passo 1: Camada de Serviço

Crie src/services/user.service.js

// Por enquanto, simulando um banco de dados em memória
const users = [{ id: 1, name: 'João' }];

class UserService {
  getAll() {
    // Lógica de negócio pura (ex: buscar no DB)
    return users;
  }

  create(userData) {
    // Lógica de negócio (ex: validar, inserir no DB)
    // Note: Recebe 'userData', não 'req.body'
    const newUser = { id: users.length + 1, ...userData };
    users.push(newUser);
    return newUser;
  }

  getById(id) {
    // Lógica pura (recebe 'id', não 'req.params')
    return users.find(u => u.id === parseInt(id));
  }
}
// Exportamos uma instância da classe
module.exports = new UserService();

Passo 2: Camada de Controle

Crie src/controllers/user.controller.js

// Importa o especialista
const userService = require('../services/user.service');

class UserController {
  // Orquestrador para buscar todos
  async getAll(req, res) {
    const users = userService.getAll(); // 1. Chama o Serviço
    res.json({ success: true, data: users }); // 2. Envia a resposta
  }

  // Orquestrador para criar um
  async create(req, res) {
    // 1. Extrai dados do 'req'
    const userData = req.body; 
    // 2. Chama o Serviço com os dados puros
    const newUser = userService.create(userData);
    // 3. Envia a resposta (Status 201 = Criado)
    res.status(201).json({ success: true, data: newUser });
  }

  // Orquestrador para buscar por ID
  async getById(req, res) {
    const id = req.params.id; // 1. Extrai o ID
    const user = userService.getById(id); // 2. Chama o Serviço
    
    if (!user) { // 3. Trata o 'não encontrado'
      return res.status(404).json({ success: false, message: 'Usuário não encontrado' });
    }
    res.json({ success: true, data: user }); // 4. Envia a resposta
  }
}
module.exports = new UserController();

Passo 3: Camada de Rota

Crie src/routes/user.routes.js

// Importamos o 'Router' do express
const { Router } = require('express');
// Importamos o 'Gerente' (Controller)
const userController = require('../controllers/user.controller');

// Criamos uma instância do Roteador
const router = Router();

// Definimos as rotas e quem as controla

// GET /users -> controlado por userController.getAll
router.get('/users', userController.getAll);

// POST /users -> controlado por userController.create
router.post('/users', userController.create);

// GET /users/:id -> controlado por userController.getById
router.get('/users/:id', userController.getById);

// Exportamos o roteador pronto
module.exports = router;

Passo 4: O index.js Limpo

Mova o código antigo para src/index.js e limpe-o:

const express = require('express');
// 1. Importamos nossas rotas de usuário
const userRoutes = require('./routes/user.routes');

const app = express();
const PORT = 3000;

// 2. Adicionamos o Middleware para ler JSON
app.use(express.json());

// 3. Usamos as rotas importadas
app.use(userRoutes);

// Rota "Hello World" (opcional)
app.get('/', (req, res) => {
  res.send('API Rodando!');
});

app.listen(PORT, () => {
  console.log(`Servidor rodando em http://localhost:${PORT}`);
});

Bônus: Prefixo de API

É uma boa prática adicionar um prefixo em todas as rotas da sua API (ex: /api ou /api/v1).

Isso ajuda a versionar sua API no futuro e a separar rotas de API de rotas de páginas web.

Em src/index.js, mude UMA linha: 

// ANTES
app.use(userRoutes);

// DEPOIS (com prefixo)
app.use('/api', userRoutes);

Agora, nossas rotas são:

  • GET /api/users
  • POST /api/users
  • GET /api/users/:id

7. Lidando com Parâmetros de Requisição

Como o Controller obtém dados do req?

Tipo 1: req.params (Parâmetros de Rota)

Usados para identificar um recurso específico. Fazem parte da URL.

Na Rota (user.routes.js):

// O ':id' é um placeholder dinâmico
router.get('/users/:id', userController.getById);

Requisição no Insomnia:

GET /api/users/1

No Controller (`user.controller.js`):

async getById(req, res) {
  const id = req.params.id; // id será "1"
  // ...
}

Tipo 2: req.body (Corpo da Requisição)

Usado para enviar dados para criar ou atualizar um recurso (em POST ou PUT).

Na Rota (user.routes.js):

router.post('/users', userController.create);

Requisição no Insomnia (com Body JSON):

POST /api/users
{ "name": "Maria", "email": "maria@senac.com" }

No Controller (`user.controller.js`):

async create(req, res) {
  // Requer app.use(express.json())
  const userData = req.body; 
  // userData será { name: "Maria", ... }
  // ...
}

Tipo 3 (Bônus): req.query (Query Params)

Usados para filtrar, ordenar ou paginar listas. Vêm APÓS O ? na URL.

Na Rota (user.routes.js):

// Usa a mesma rota GET /users
router.get('/users', userController.getAll);

Requisição no Insomnia:

GET /api/users?status=ativo&sort=nome

No Controller (`user.controller.js`):

async getAll(req, res) {
  const status = req.query.status; // status será "ativo"
  const sort = req.query.sort;   // sort será "nome"
  // ... lógica para filtrar/ordenar ...
}

Resumo dos Parâmetros

Tipo Onde Vem? Uso Comum
req.params Na Rota (/users/:id) Identificar recurso (GET, PUT, DELETE)
req.body Corpo da Requisição (JSON) Enviar dados (POST, PUT)
req.query Após o ? (?filtro=...) Filtrar, Ordenar, Paginar (GET)
