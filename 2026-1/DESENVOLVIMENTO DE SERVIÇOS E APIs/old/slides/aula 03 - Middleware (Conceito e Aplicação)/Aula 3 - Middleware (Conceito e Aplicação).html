O Problema: Preocupações Transversais

O que acontece se precisarmos executar uma lógica em VÁRIAS ou TODAS as nossas rotas?

  • ? "Quero registrar (logar) no console toda requisição que chega na API."
  • ? "Quero validar se o `name` e `email` existem ANTES de chegar no Controller."
  • ? "Quero verificar se o usuário está autenticado (logado) antes de permitir que ele acesse a rota."

A Solução Ruim: Repetir Código

Poderíamos colocar essa lógica em cada método do Controller... 

class UserController {
  async create(req, res) {
    // 1. Logar a requisição
    console.log("Requisição recebida!"); // <--- REPETIDO
    
    // 2. Validar o body
    const { name, email } = req.body;
    if (!name) { // <--- REPETIDO
      return res.status(400).json(...);
    }
    // ... resto da lógica ...
  }

  async update(req, res) {
    // 1. Logar a requisição
    console.log("Requisição recebida!"); // <--- REPETIDO

    // 2. Validar o body... // <--- REPETIDO
  }
}

Isso viola o princípio DRY (Don't Repeat Yourself)!

A Solução Elegante: Middleware

Middleware é um conceito de design que nos permite "interceptar" uma requisição ANTES que ela chegue ao seu destino final (o Controller).

É um "checkpoint" ou um "porteiro" que fica no meio do caminho.

A requisição é "inspecionada" pelo Middleware antes de continuar.

O Princípio DRY

"Don't Repeat Yourself" (Não se Repita)

Definição:

"Cada parte do conhecimento (lógica, regra de negócio) deve ter uma representação única, inequívoca e autoritativa dentro de um sistema."

O Problema (Anti-DRY): Se a lógica de validação de email está em 5 controllers, e você precisa atualizá-la, você tem que alterar em 5 lugares. Se esquecer de 1, você tem um bug.

A Solução (DRY com Middlewares): Você cria a lógica de validação UMA VEZ (no middleware) e a "anexa" em 5 rotas. Para atualizar, você altera em UM SÓ LUGAR.

1. Teoria: O que é um Middleware?

O "Porteiro" 💂

No Express, um Middleware é simplesmente uma função.

É uma função que tem acesso:

  • Ao objeto de Requisição (`req`).
  • Ao objeto de Resposta (`res`).
  • À PRÓXIMA função no ciclo da requisição (next).

O "Porteiro" em Ação

Quando uma requisição chega, o "Porteiro" (Middleware) pode:

1. Apenas observar e deixar passar:
"Vi que a requisição chegou às 10:05. Pode continuar."
(Ex: Logger)

2. Modificar e deixar passar:
"Vi que você não tem um 'crachá'. Aqui está um crachá de 'Visitante'. Pode continuar."
(Ex: Adicionar `req.user` em um middleware de auth)

3. Barrar a entrada:
"Você não tem convite. Não pode entrar. Dê meia volta."
(Ex: Validação ou Autenticação falhou)

2. A Assinatura Mágica: `(req, res, next)`

Toda função middleware (padrão) tem esta assinatura: 

function meuMiddleware(req, res, next) {
  // 1. O que fazer com a requisição?
  
  // 2. Terminei? Deixo o próximo trabalhar.
  next();
}

  • req (Requisição)

    O mesmo `req` do Controller. Podemos ler (req.body) ou até adicionar dados (req.userId = 123).

  • res (Resposta)

    O mesmo `res` do Controller. O middleware pode usá-lo para encerrar a requisição (ex: res.status(401).send()).

O Conceito Mais Importante: next()

`next()` é uma função que, ao ser chamada, passa o controle para o próximo middleware na "esteira" de execução.

✔ Se você chamar `next()`:
O Express procura o próximo middleware ou o Controller final da rota e o executa.

❌ Se você NÃO chamar `next()` (e não enviar resposta):
A requisição fica "presa" (pendurada). O cliente (Insomnia) dará "Timeout".

! Se você enviar uma resposta (ex: `res.json()`):
Você NÃO DEVE chamar `next()`. A requisição já foi encerrada e respondida.

3. A "Esteira" de Middlewares

Pense em uma linha de montagem. A requisição (`req`) é o produto, e cada middleware é um "braço robótico" que faz algo com ela.

app.use(mw1) ➔ app.use(mw2) ➔ router.post('/', mw3, controller)

A Ordem de Declaração IMPORTA!

Middlewares são executados na ordem exata em que são declarados no código (com `app.use` ou `router.use`).

Exemplo CORRETO:

// 1. Primeiro, ensinamos o Express a ler JSON
app.use(express.json());
// 2. Agora, o validador pode LER o req.body
app.use(meuValidador);

Exemplo ERRADO:

// 1. O validador tenta ler req.body.
app.use(meuValidador); // CRASH! req.body é undefined aqui
// 2. Só agora o Express aprende a ler JSON. Tarde demais.
app.use(express.json());

4. Classificando (Por Nível de Aplicação)

A forma mais importante de classificar é por COMO ele é aplicado:

  • Nível de Aplicação (Global):

    Executa em TODAS as requisições.
    Ex: app.use(meuLogger);

  • Nível de Roteador:

    Executa em todas as rotas de um roteador específico.
    Ex: userRoutes.use(checkAuth);

  • Nível de Rota (Específico):

    Executa apenas em uma rota.
    Ex: router.post('/', validateUser, controller);

Classificando (Por Origem vs. Aplicação)

A "Origem" (de onde vem) e a "Aplicação" (como usamos) se misturam.

1. Nativos (Built-in)

Vêm com o Express. (Ex: express.json())

Como usamos: Quase sempre em Nível de Aplicação (Global).
app.use(express.json());

2. De Terceiros (Third-party)

Instalados via NPM (Ex: cors, helmet)

Como usamos: Geralmente em Nível de Aplicação (Global).
app.use(cors());

3. Customizados (Nós que fazemos!)

Nossas próprias funções (Ex: logger, validateUser)

Como usamos: Onde quisermos!
· app.use(logger) -> Global
· router.post('/', validateUser, ...) -> Específico de Rota

Tipo Especial: Error Handling

(Vamos implementar na Aula 4, mas é importante conhecer agora).

Existe um tipo especial de middleware para capturar erros.

Ele tem uma assinatura diferente, com 4 argumentos: 

// O 'err' vem primeiro!
function meuTratadorDeErros(err, req, res, next) {
  console.error(err); // Loga o erro
  res.status(500).json({ message: 'Algo deu errado!' });
}

Ele é ativado quando chamamos next(new Error()).

6. Exemplo 1: Logger (Global)

Objetivo: Registrar no console TODA requisição que chega na API, mostrando o método, a URL e a data.

Tipo: Customizado, aplicado em Nível de Aplicação (Global).

Ação do Porteiro: "Anotar e deixar passar".

Código: Logger

Crie src/middlewares/logger.middleware.js

function logger(req, res, next) {
  // 1. Obtém a data/hora atual em formato ISO
  const timestamp = new Date().toISOString();
  
  // 2. Obtém o método HTTP (GET, POST, etc.)
  const method = req.method;
  
  // 3. Obtém a URL original que o cliente pediu
  const url = req.originalUrl;

  // 4. Loga a informação no console do servidor
  console.log(
    `[${timestamp}] ${method} ${url}`
  );

  // 5. ESSENCIAL: Passa o controle para o próximo
  next();
}

module.exports = logger;

Aplicando o Logger

Para que ele rode em TODAS as rotas, o aplicamos no src/index.js com app.use(). 

const express = require('express');
const userRoutes = require('./routes/user.routes');
// 1. Importamos o logger
const logger = require('./middlewares/logger.middleware');

const app = express();
const PORT = 3000;

// --- ORDEM DOS MIDDLEWARES ---
// 1. Primeiro, o parser de JSON
app.use(express.json());

// 2. Segundo, nosso Logger (global)
app.use(logger);
// --- FIM DOS MIDDLEWARES GLOBAIS ---

// 3. Por último, as Rotas
app.use('/api', userRoutes);

app.listen(...);

Testando o Logger

1. Verifique se o servidor está rodando com npm run dev.

2. Abra o Insomnia.

3. Faça QUALQUER requisição (GET /api/users, GET /api/users/1, etc).

4. Olhe o seu Terminal (onde o `nodemon` está rodando).

Você deve ver os logs aparecendo: 

[2025-11-02T17:10:30.123Z] GET /api/users
[2025-11-02T17:10:35.456Z] GET /api/users/1

7. Exemplo 2: Validação (Por Rota)

Objetivo: Garantir que, ao criar um usuário (POST /api/users), os campos `name` e `email` obrigatoriamente existam no `req.body`.

Tipo: Customizado, aplicado em Nível de Rota (Específico).

Ação do Porteiro: "Verificar o convite. Se estiver OK, deixe passar. Se não, barre a entrada (400 Bad Request)."

Código: Validação

Crie src/middlewares/validation.middleware.js

function validateUserCreation(req, res, next) {
  // 1. Puxamos 'name' e 'email' do req.body
  // (Isso só funciona se app.use(express.json()) veio antes)
  const { name, email } = req.body;

  // 2. Verificamos se algum deles está faltando
  if (!name || !email) {
    // 3. Se faltar: BARRAMOS A ENTRADA
    // Enviamos uma resposta de erro (400 Bad Request)
    // E NÃO CHAMAMOS next()!
    return res.status(400).json({
      success: false,
      message: 'Nome e email são obrigatórios.'
    });
  }

  // 4. Se passou na validação: DEIXE PASSAR
  // Chamamos next() para ir para o Controller
  next();
}

module.exports = { validateUserCreation };

Aplicando a Validação

Não queremos validar isso em TODAS as rotas, apenas na de criação. Por isso, aplicamos "em linha" no arquivo src/routes/user.routes.js. 

const { Router } = require('express');
const userController = require('../controllers/user.controller');
// 1. Importamos nosso validador
const { validateUserCreation } = require('../middlewares/validation.middleware');

const router = Router();

// Rota GET (sem validação de body)
router.get('/users', userController.getAll);

// 2. Adicionamos o middleware ANTES do controller
router.post(
  '/users', 
  validateUserCreation,  // <-- ELE RODA PRIMEIRO
  userController.create  // <-- SÓ RODA SE O PRIMEIRO CHAMAR next()
);

router.get('/users/:id', userController.getById);

module.exports = router;

O Fluxo de Validação (Visual)

Quando uma requisição POST /api/users chega:

1. Requisição ➔ express.json() (Lê o Body)

2. ➔ logger() (Loga no console) ➔ `next()`

3. ➔ Roteador acha POST /api/users

4. ➔ validateUserCreation() (Verifica `req.body`)

↳ Falha: res.status(400).json() (Controller NUNCA é chamado)

↳ Sucesso: next()

5. ➔ userController.create() (Controller finalmente trabalha)

1. No Insomnia, crie uma requisição POST /api/users.

2. No Body (JSON), envie dados incompletos: 

{
  "name": "Carlos"
  // Faltando o "email"
}

3. Clique em "Send".

Resultado Esperado:

  "success": false,
  "message": "Nome e email são obrigatórios."
}

O Controller nem foi acionado. O Middleware nos protegeu.

Testando a Validação (Cenário 2: Sucesso)

1. Na mesma requisição do Insomnia...

2. No Body (JSON), envie dados completos:

Resultado Esperado:

Bônus: Middlewares de Terceiros

Se você tentar acessar sua API de um site (React, Angular) em localhost:3001, o navegador irá BLOQUEAR a requisição por segurança (Política de Mesma Origem - CORS).

Para consertar isso, usamos um middleware de terceiro, o cors. 

# 1. Instalar
npm install cors

// 2. Usar (no index.js, globalmente)
const cors = require('cors');
app.use(cors()); // Permite acesso de qualquer origem
app.use(express.json());
app.use(logger);
...

Diagrama de Fluxo Completo (Até agora)

Requisição: POST /api/users com Body `{ "name": "Ana" }`

  1. app.use(express.json()) ➔ Lê o Body. Chama `next()`.
  2. app.use(logger()) ➔ Loga no console. Chama `next()`.
  3. app.use('/api', userRoutes) ➔ Direciona para o Roteador.
  4. router.post('/users', ...) ➔ Encontra a rota.
  5. validateUserCreation() ➔ Validação falha (falta email).
  6. return res.status(400).json(...) ➔ Resposta é enviada.
  7. O fluxo é INTERROMPIDO. O Controller NÃO é chamado.

Revisão: O que é "Middleware"?

O nome "Middleware" (Software do Meio) faz sentido:

É um software que fica no meio do "pedido" (Requisição) e do "processamento final" (Controller).

Eles formam uma cadeia (chain) de processamento. São a base de como aplicações Express.js gerenciam autenticação, validação, logging e muito mais.
