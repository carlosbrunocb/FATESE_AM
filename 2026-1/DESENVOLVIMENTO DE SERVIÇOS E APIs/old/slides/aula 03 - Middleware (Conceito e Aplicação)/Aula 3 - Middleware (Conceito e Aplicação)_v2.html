O Problema: Preocupações Transversais

O que acontece se precisarmos executar uma lógica em VÁRIAS ou TODAS as nossas rotas?

  • ? "Quero registrar (logar) no console toda requisição que chega na API."
  • ? "Quero validar se o name e email existem ANTES de chegar no Controller."
  • ? "Quero verificar se o usuário está autenticado (logado) antes de permitir que ele acesse a rota."

A Solução Ruim: Repetir Código

Poderíamos colocar essa lógica em cada método do Controller... 

class UserController {
  async create(req, res) {
    // 1. Logar a requisição
    console.log("Requisição recebida!"); // <--- REPETIDO
    
    // 2. Validar o body
    const { name, email } = req.body;
    if (!name) { // <--- REPETIDO
      return res.status(400).json(...);
    }
    // ... resto da lógica ...
  }

  async update(req, res) {
    // 1. Logar a requisição
    console.log("Requisição recebida!"); // <--- REPETIDO

    // 2. Validar o body... // <--- REPETIDO
  }
}

Isso viola o princípio DRY (Don't Repeat Yourself)!

A Solução Elegante: Middleware

Middleware é um conceito de design que nos permite "interceptar" uma requisição ANTES que ela chegue ao seu destino final (o Controller).

É um "checkpoint" ou um "porteiro" que fica no meio do caminho.

A requisição é "inspecionada" pelo Middleware antes de continuar.

O Princípio DRY

"Don't Repeat Yourself" (Não se Repita)

Definição:

"Cada parte do conhecimento (lógica, regra de negócio) deve ter uma representação única, inequívoca e autoritativa dentro de um sistema."

O Problema (Anti-DRY): Se a lógica de validação de email está em 5 controllers, e você precisa atualizá-la, você tem que alterar em 5 lugares. Se esquecer de 1, você tem um bug.

A Solução (DRY com Middlewares): Você cria a lógica de validação UMA VEZ (no middleware) e a "anexa" em 5 rotas. Para atualizar, você altera em UM SÓ LUGAR.

1. Teoria: O que é um Middleware?

O "Porteiro" 💂

No Express, um Middleware é simplesmente uma função.

É uma função que tem acesso:

  • Ao objeto de Requisição (req).
  • Ao objeto de Resposta (res).
  • À PRÓXIMA função no ciclo da requisição (next).

O que é Middleware? (Definição)

Middlewares são funções que têm acesso aos objetos de requisição (req), resposta (res) e à próxima função de middleware (next) no ciclo de requisição-resposta da aplicação. Eles atuam como intermediários entre a chegada da requisição e o handler final.

  • Funções intermediárias que interceptam e processam requisições HTTP antes de alcançarem os controllers finais.

  • Acesso total aos objetos req (requisição), res (resposta) e next (próxima função no ciclo).

  • Podem executar código, modificar req/res, encerrar o ciclo ou chamar o próximo middleware na pilha.

  • Formam um pipeline de processamento que é executado sequencialmente na ordem de registro.

Analogia: "Porteiros" ou "Checkpoints"

Exemplos comuns:

  • Autenticação
  • Validação de dados
  • Logging
  • Segurança (CORS)

Quando usar Middlewares

Middlewares são ideais para processar requisições de forma consistente e modular. Conheça os principais casos de uso e quando aplicá-los:

Autenticação e Autorização

Validar tokens JWT, verificar permissões e proteger rotas baseadas em roles de usuário.

Logging e Auditoria

Registrar detalhes de cada requisição (método, URL, IP, tempo de resposta) para monitoramento e depuração.

Validação de Dados

Validar e sanitizar dados de entrada antes que cheguem aos controllers (usando Joi, Zod, Yup).

Parsing de Dados

Processar dados recebidos (JSON, formulários, multipart/form-data) em formatos utilizáveis.

CORS e Segurança

Gerenciar Cross-Origin Resource Sharing, adicionar headers de segurança e proteção contra ataques comuns.

Compressão e Performance

Comprimir respostas, cachear resultados e otimizar o desempenho da API.

Rate Limiting

Limitar número de requisições por cliente para prevenir abusos e garantir disponibilidade do serviço.

Tratamento de Erros

Capturar e processar erros de forma centralizada, garantindo respostas consistentes e seguras.

O "Porteiro" em Ação

Quando uma requisição chega, o "Porteiro" (Middleware) pode:

1. Apenas observar e deixar passar:

"Vi que a requisição chegou às 10:05. Pode continuar."

Ex: Logger
2. Modificar e deixar passar:

"Vi que você não tem um 'crachá'. Aqui está um crachá de 'Visitante'. Pode continuar."

Ex: Adicionar req.user em um middleware de auth
3. Barrar a entrada:

"Você não tem convite. Não pode entrar. Dê meia volta."

Ex: Validação ou Autenticação falhou

Middlewares são como porteiros que verificam, processam e encaminham requisições antes de chegarem ao destino final.

Cada middleware pode: (1) executar verificações, (2) modificar a requisição/resposta, (3) encerrar o ciclo ou (4) passar para o próximo com next().

Se um porteiro (middleware) detecta um problema, ele pode "barrar" a requisição sem chamar next(), enviando uma resposta de erro.

Ciclo Requisição-Resposta

Cliente

Servidor Express

req (request)
parse
auth
validate
controller
res (response)

Fluxo: O cliente envia uma requisição que passa por vários middlewares antes de chegar ao controller.

Objeto req: Contém informações (body, params, query) e é enriquecido ao passar pelos middlewares.

Objeto res: Permite enviar a resposta (com res.json(), res.send(), res.status()). Quando um método de resposta é chamado, o ciclo é encerrado.

Arquitetura do Express e Middlewares

Application-Level Middleware

Escopo Global - Todas as requisições

app.use(middleware)

Router-Level Middleware

Escopo de Roteador - Grupo de rotas

router.use(middleware)

Route Handler

Controlador final

(req, res) => { ... }

Resposta ao Cliente

Retorno da solicitação

res.json(...)

Ordem importa: Middlewares são executados na sequência exata em que são registrados no código.

Pipeline de processamento: Uma requisição atravessa cada middleware na ordem até encontrar uma resposta ou o handler final.

Função next(): Cada middleware decide se encaminha a requisição para o próximo da cadeia ou encerra o ciclo com uma resposta.

Anatomia de um Middleware

function meuMiddleware(req, res, next) {
  // 1) Executa alguma lógica (log, auth, parse, validação...)
  const start = new Date().getTime();
  console.log(`Recebida requisição ${req.method} para ${req.url}`);

  // 2) Pode alterar req/res: ex. req.user = {...}
  req.requestTime = start;
  res.set('X-Processed-By', 'Meu Middleware');

  // 3) Decide: encerrar (res.status(...).json(...)) OU seguir
  if (req.path === '/proibido') {
    return res.status(403).json({
      success: false,
      message: 'Acesso negado'
    });
  }

  next(); // Importante para continuar o pipeline
}

// Registro (app-level)
app.use(meuMiddleware);

// Registro (router-level)
router.use(meuMiddleware);

Componentes essenciais:

Função com 3 parâmetros: req, res e next.

next() é crucial - sem ele a requisição fica "presa" no middleware.

Um middleware pode encerrar o ciclo com uma resposta OU chamar next().

2. A Assinatura Mágica: (req, res, next)

Toda função middleware (padrão) tem esta assinatura: 

function meuMiddleware(req, res, next) {
  // 1. O que fazer com a requisição?
  
  // 2. Terminei? Deixo o próximo trabalhar.
  next();
}

  • req (Requisição)

    O mesmo req do Controller. Podemos ler (req.body) ou até adicionar dados (req.userId = 123).

  • res (Resposta)

    O mesmo res do Controller. O middleware pode usá-lo para encerrar a requisição (ex: res.status(401).send()).

O Objeto Request (req)

O objeto request (req) representa a requisição HTTP enviada pelo cliente. Ele contém toda a informação relevante como cabeçalhos, corpo, parâmetros, URL e outros metadados que chegam ao servidor Express.

  • req.body: Contém dados do corpo da requisição (POST/PUT). Requer middleware de parsing (express.json()).

  • req.params: Parâmetros de rota definidos na URL (ex: /users/:id).

  • req.query: Objeto contendo valores da query string (ex: ?page=2).

  • req.headers: Todos os cabeçalhos HTTP enviados (ex: Authorization).

  • Outros: req.method (GET/POST), req.url, req.ip, req.path.

Exemplo de uso:

// Obtendo dados da requisição
const { id } = req.params;
const { page, limit } = req.query;
const { name, email } = req.body;
const token = req.headers.authorization;
const clientIP = req.ip;

// No middleware você pode modificar req
req.user = { id: '123', role: 'admin' };

O Objeto Response (res)

O objeto response (res) é a segunda parte fundamental do middleware e representa a resposta HTTP que o servidor Express envia de volta ao cliente.

  • res.status(codigo): Define o código de status HTTP (200, 404, 500, etc).

  • res.json(dados): Envia uma resposta JSON formatada.

  • res.send(conteúdo): Envia resposta (texto, HTML, etc).

  • res.set(header, valor): Define cabeçalhos HTTP.

  • res.redirect(url): Redireciona o cliente.

// Resposta de sucesso com dados
res.status(200).json({
  "success": true,
  "data": usuario
});

// Resposta de erro
res.status(404).json({
  "success": false,
  "message": 'Usuário não encontrado'
});

// IMPORTANTE: Após enviar uma
// resposta, não chame next()!

A função next() em detalhes

A função next() é o mecanismo que controla o fluxo entre middlewares no Express, permitindo passar a execução para o próximo da pilha ou encaminhar erros.

  • Função básica: Passa o controle para o próximo middleware registrado na pilha de execução.

  • next(err): Pula todos os middlewares regulares e passa para o primeiro middleware de tratamento de erro.

  • Não chamar next(): Se não chamar next() nem enviar resposta, a requisição ficará pendente até timeout.

  • Boas práticas: Sempre chamar next() apenas uma vez; se enviou resposta, não chame next().

Fluxo de Execução com next()

Middleware 1

Middleware 2

Controller

Error Handler

next() → Fluxo normal

next(err) → Fluxo de erro

O Conceito Mais Importante: next()

next() é uma função que, ao ser chamada, passa o controle para o próximo middleware na "esteira" de execução.

✔ Se você chamar next():
O Express procura o próximo middleware ou o Controller final da rota e o executa.
❌ Se você NÃO chamar next()
(e não enviar resposta):
A requisição fica "presa" (pendurada). O cliente (Insomnia) dará "Timeout".
! Se você enviar uma resposta
(ex: res.json()):
Você NÃO DEVE chamar next(). A requisição já foi encerrada e respondida.

3. A "Esteira" de Middlewares

Pense em uma linha de montagem. A requisição (req) é o produto, e cada middleware é um "braço robótico" que faz algo com ela.

app.use(mw1) ➔ app.use(mw2) ➔ router.post('/', mw3, controller)

A Ordem de Declaração IMPORTA!

Middlewares são executados na ordem exata em que são declarados no código (com `app.use` ou `router.use`).

Exemplo CORRETO:

// 1. Primeiro, ensinamos o Express a ler JSON
app.use(express.json());
// 2. Agora, o validador pode LER o req.body
app.use(meuValidador);

Exemplo ERRADO: 

// 1. O validador tenta ler req.body.
app.use(meuValidador); // CRASH! req.body é undefined aqui
// 2. Só agora o Express aprende a ler JSON. Tarde demais.
app.use(express.json());

Fluxo de Execução Sequencial

const express = require('express');
const app = express();

// 1. Executa primeiro (parser)
app.use(express.json()); 

// 2. Executa segundo (logger global)
app.use((req, res, next) => {
  console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
  next();
});

// 3. Executa terceiro (apenas para rotas /api/*)
app.use('/api', apiRouter);

// 4. Executa apenas para /health (fora de /api)
app.get('/health', (req, res) => res.json({ ok: true }));

// 5. Executa apenas em caso de erro (sempre por último)
app.use((err, req, res, next) => {
  console.error('Erro:', err);
  res.status(500).json({ error: 'Erro interno' });
});

Fluxo de execução para requisição GET /api/users:

1. express.json(): parseia o corpo da requisição (se existir)

2. logger: registra a requisição no console

3. apiRouter: contém as rotas /api/* e manipula a requisição

A rota /health não passa pelo apiRouter, pois está em outro caminho

IMPORTANTE: A ordem de registro determina a ordem de execução!

4. Classificando (Por Nível de Aplicação)

A forma mais importante de classificar é por COMO ele é aplicado:

  • Nível de Aplicação (Global):

    Executa em TODAS as requisições.
    Ex: app.use(meuLogger);

  • Nível de Roteador:

    Executa em todas as rotas de um roteador específico.
    Ex: userRoutes.use(checkAuth);

  • Nível de Rota (Específico):

    Executa apenas em uma rota.
    Ex: router.post('/', validateUser, controller);

Classificando (Por Origem vs. Aplicação)

A "Origem" (de onde vem) e a "Aplicação" (como usamos) se misturam.

1. Nativos (Built-in)

Vêm com o Express. (Ex: express.json())

Como usamos: Quase sempre em Nível de Aplicação (Global).
app.use(express.json());

2. De Terceiros (Third-party)

Instalados via NPM (Ex: cors, helmet)

Como usamos: Geralmente em Nível de Aplicação (Global).
app.use(cors());

3. Customizados (Nós que fazemos!)

Nossas próprias funções (Ex: logger, validateUser)

Como usamos: Onde quisermos!
· app.use(logger) -> Global
· router.post('/', validateUser, ...) -> Específico de Rota

Tipo Especial: Error Handling

(Vamos implementar na Aula 4, mas é importante conhecer agora).

Existe um tipo especial de middleware para capturar erros.

Ele tem uma assinatura diferente, com 4 argumentos: 

// O 'err' vem primeiro!
function meuTratadorDeErros(err, req, res, next) {
  console.error(err); // Loga o erro
  res.status(500).json({ message: 'Algo deu errado!' });
}

Ele é ativado quando chamamos next(new Error()).

6. Exemplo 1: Logger (Global)

Objetivo: Registrar no console TODA requisição que chega na API, mostrando o método, a URL e a data.

Tipo: Customizado, aplicado em Nível de Aplicação (Global).

Ação do Porteiro: "Anotar e deixar passar".

Código: Logger

Crie src/middlewares/logger.middleware.js

function logger(req, res, next) {
  // 1. Obtém a data/hora atual em formato ISO
  const timestamp = new Date().toISOString();
  
  // 2. Obtém o método HTTP (GET, POST, etc.)
  const method = req.method;
  
  // 3. Obtém a URL original que o cliente pediu
  const url = req.originalUrl;

  // 4. Loga a informação no console do servidor
  console.log(
    `[${timestamp}] ${method} ${url}`
  );

  // 5. ESSENCIAL: Passa o controle para o próximo
  next();
}

module.exports = logger;

Aplicando o Logger

Para que ele rode em TODAS as rotas, o aplicamos no src/index.js com app.use(). 

const express = require('express');
const userRoutes = require('./routes/user.routes');
// 1. Importamos o logger
const logger = require('./middlewares/logger.middleware');

const app = express();
const PORT = 3000;

// --- ORDEM DOS MIDDLEWARES ---
// 1. Primeiro, o parser de JSON
app.use(express.json());

// 2. Segundo, nosso Logger (global)
app.use(logger);
// --- FIM DOS MIDDLEWARES GLOBAIS ---

// 3. Por último, as Rotas
app.use('/api', userRoutes);

app.listen(...);

7. Exemplo 2: Validação (Por Rota)

Objetivo: Garantir que, ao criar um usuário (POST /api/users), os campos name e email obrigatoriamente existam no req.body.

Tipo: Customizado, aplicado em Nível de Rota (Específico).

Ação do Porteiro: "Verificar o convite. Se estiver OK, deixe passar. Se não, barre a entrada (400 Bad Request)."

Código: Validação

Crie src/middlewares/validation.middleware.js

function validateUserCreation(req, res, next) {
  // 1. Puxamos 'name' e 'email' do req.body
  // (Isso só funciona se app.use(express.json()) veio antes)
  const { name, email } = req.body;

  // 2. Verificamos se algum deles está faltando
  if (!name || !email) {
    // 3. Se faltar: BARRAMOS A ENTRADA
    // Enviamos uma resposta de erro (400 Bad Request)
    // E NÃO CHAMAMOS next()!
    return res.status(400).json({
      success: false,
      message: 'Nome e email são obrigatórios.'
    });
  }

  // 4. Se passou na validação: DEIXE PASSAR
  // Chamamos next() para ir para o Controller
  next();
}

module.exports = { validateUserCreation };

Aplicando a Validação

Não queremos validar isso em TODAS as rotas, apenas na de criação. Por isso, aplicamos "em linha" no arquivo src/routes/user.routes.js. 

const { Router } = require('express');
const userController = require('../controllers/user.controller');
// 1. Importamos nosso validador
const { validateUserCreation } = require('../middlewares/validation.middleware');

const router = Router();

// Rota GET (sem validação de body)
router.get('/users', userController.getAll);

// 2. Adicionamos o middleware ANTES do controller
router.post(
  '/users', 
  validateUserCreation,  // <-- ELE RODA PRIMEIRO
  userController.create  // <-- SÓ RODA SE O PRIMEIRO CHAMAR next()
);

router.get('/users/:id', userController.getById);

module.exports = router;

O Fluxo de Validação (Visual)

Quando uma requisição POST /api/users chega:

1. Requisição ➔ express.json() (Lê o Body)

2. ➔ logger() (Loga no console) ➔ next()

3. ➔ Roteador acha POST /api/users

4. ➔ validateUserCreation() (Verifica req.body)

↳ Falha: res.status(400).json() (Controller NUNCA é chamado)

↳ Sucesso: next()

5. ➔ userController.create() (Controller finalmente trabalha)

Bônus: Middlewares de Terceiros

Revisão: O que é "Middleware"?

O nome "Middleware" (Software do Meio) faz sentido:

É um software que fica no meio do "pedido" (Requisição) e do "processamento final" (Controller).

Eles formam uma cadeia (chain) de processamento. São a base de como aplicações Express.js gerenciam autenticação, validação, logging e muito mais.
