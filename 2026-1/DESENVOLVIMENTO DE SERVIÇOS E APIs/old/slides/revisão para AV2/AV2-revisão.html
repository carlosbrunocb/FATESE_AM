Fundamentos de APIs e Node.js

O que é Node.js?

Não é uma linguagem (é JS) e não é um framework. É um Ambiente de Execução (Runtime) construído sobre o motor V8 do Chrome. Permite rodar JavaScript no servidor (Back-end) com acesso a arquivos, redes e banco de dados.

  • Non-blocking I/O: Não espera operações lentas terminarem, usa callbacks/promises.
  • Single-threaded: Executa uma coisa de cada vez, mas gerencia concorrência pelo Event Loop.

Conceito REST

REST (Representational State Transfer) é um estilo arquitetural. Principais regras:

  • Stateless: O servidor não guarda estado do cliente entre requisições.
  • Cliente-Servidor: Separação clara de responsabilidades.

Verbos HTTP (CRUD)

GET Ler dados (Read)
POST Criar dados (Create)
PUT Atualizar dados (Update)
DELETE Remover dados (Delete)

Setup Básico:

npm init -y npm install express

Arquitetura em Camadas (SoC)

Conceito Chave: Separação de Responsabilidades (Separation of Concerns). Evitar o "arquivo único gigante" para facilitar manutenção e testes.

1. Rotas (Routes)

O "Recepcionista". Define URLs e verbos. Não tem lógica, apenas direciona para o Controller.

2. Controller

O "Gerente". Recebe req e res. Extrai dados, chama o Service e responde ao cliente.

3. Service

O "Especialista". Contém a Regra de Negócio pura. Acessa o banco. Nunca vê req/res.

Recebendo Dados (Request)

Tipo Uso Exemplo
req.params Identificar recurso na URL /users/:id
req.body Enviar dados (JSON) em POST/PUT { "nome": "Ana" }
req.query Filtros e paginação (após o ?) /users?page=2

* Para usar req.body, é obrigatório: app.use(express.json())

3

Middlewares

São funções que interceptam a requisição antes de chegar ao controller final. Funcionam como "Porteiros" ou uma "Esteira de Fábrica".

Assinatura Mágica

Todo middleware padrão recebe 3 argumentos:

function meuMiddleware(req, res, next) { // Faz algo... next(); // OBRIGATÓRIO para continuar! }

O que o Middleware pode fazer?

  • Executar código.
  • Modificar req e res (ex: add usuário logado).
  • Encerrar o ciclo (ex: retornar erro 403).
  • Chamar o próximo com next().

Tipos Comuns

  • Global: app.use(logger) (Roda em tudo)
  • Rota Específica: router.post('/', valida, controller)
  • Terceiros: cors, helmet.
Atenção: A ordem de declaração importa! Se você definir o middleware depois da rota, ele não será executado para aquela rota.

Tratamento de Erros

O objetivo é centralizar os erros para evitar try...catch repetitivos e manter respostas consistentes.

O Middleware de Erro

O Express identifica um middleware de erro pela quantidade de argumentos. Ele DEVE ter 4 argumentos:

// (err, req, res, next) function errorHandler(err, req, res, next) { const status = err.statusCode || 500; res.status(status).json({ success: false, message: err.message }); }

Regra de Ouro da Ordem

O Middleware de Erro deve ser registrado POR ÚLTIMO no app.js / index.js, depois de todas as rotas.

1. app.use(express.json()) 2. app.use(rotas) 3. app.use(errorHandler) // ÚLTIMO!

Como disparar o erro?

Nos controllers, usamos o next(error) dentro do bloco catch (ou através de bibliotecas como express-async-errors ou wrappers).

try { // logica... } catch (err) { next(err); // Pula para o middleware de erro }
400Bad Request
401Unauthorized
404Not Found
500Server Error
