Feature Principal vs Secundárias (MVP)

Defina o núcleo do negócio antes de “perfumarias”

Projeto Integrador I

Diagrama do Escopo (Alvo do MVP)

Centro = o que faz o sistema existir

Anti-NASA

Secundárias / Futuro

“Bom ter”, não essencial no 1º release

Essenciais do MVP

Suportam o fluxo principal ponta a ponta

Feature Principal (Killer Feature)

Se tirar, o sistema perde a razão de existir

Como identificar a Feature Principal

Pergunta: “Qual problema central o usuário resolve aqui?”

Dica prática para o PI I

No MVP, garanta o fluxo principal funcionando antes de extras.

Definições (base teórica)

Feature Principal (Killer Feature)

É o coração do projeto. Sem isso, não existe produto.

“Se eu remover isso, ainda faz sentido?”

Evite ter 3 “principais”

Features Secundárias

Agregam valor, mas não são vitais para o primeiro lançamento (MVP).

Exemplos típicos

foto de perfil • badges • chat • temas

Critério

“O usuário vive sem isso?”

Exemplos rápidos (teóricos)

iFood

Principal: pedir comida

Secundária: chat com suporte

Uber

Principal: solicitar viagem

Secundária: foto de perfil

Um MVP forte é aquele que entrega a Feature Principal com qualidade e com poucas dependências.

Saída: 1–2 Features Principais definidas + lista mínima para o MVP.
