Requisitos Funcionais (RF)

O QUE o sistema deve fazer (ações e comportamentos)

Projeto Integrador I

Conceito (base teórica)

RF descrevem as funcionalidades do sistema: ações que usuários ou administradores conseguem executar. Devem ser claros, curtos e testáveis.

Validação: “Consigo testar isso no sistema?” (sim/não)

Dica de redação (padrão recomendado)

Comece com: “O sistema deve permitir…” e complete com um verbo de ação + objeto (cadastro, consulta, edição, exclusão, geração de relatório).

Evite: “O sistema será moderno e rápido” (isso é RNF)

Foque em fluxo: entrada → processamento → saída

Formato (ID + frase) e exemplos práticos

Use identificadores para rastrear e discutir em equipe: RF01, RF02, RF03…

RF01

O sistema deve permitir cadastrar usuários (admin).

RF02

O sistema deve permitir login de administrador.

RF03

O sistema deve permitir listar e filtrar registros.

RF04

O sistema deve permitir gerar relatório mensal simples.

Se não cabe em 1 frase clara, reescreva (ou quebre em RF menores).

Mini-diagrama: RF no fluxo do MVP

Traduza a ideia em passos executáveis

1

Identifique a Feature Principal

“Sem isso, o sistema não existe”.

2

Quebre em RF (passos do usuário)

Cadastro, consulta, atualização, exclusão, relatórios.

3

Defina RNF para “travar” o projeto

Plataforma, desempenho, segurança, tecnologia.

Regra de ouro

RF são o que o sistema faz. Se você estiver falando de tempo, segurança ou tecnologia, provavelmente é RNF.

Saída desta etapa: uma lista mínima de RF numerados para virar User Stories na Unidade II.

Dica: escreva RF com verbos e um resultado observável (o que aparece na tela / o que é salvo).
