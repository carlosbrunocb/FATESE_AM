MoSCoW: Must/Should/Could/Won’t

Exemplo de conversão (RF → User Story)

RF: “O sistema deve permitir que o administrador cadastre horários de recolha.”

User Story: Como administrador, eu quero cadastrar horários para que os moradores consultem o cronograma.

Critério de aceite: ao salvar, o horário aparece na listagem do cronograma e pode ser editado/excluído.

Foque no essencial do MVP

Próximos passos

O que fazer após sair desta aula

1) Fechar RF/RNF no escopo

Requisitos numerados, curtos e testáveis.

2) Criar User Stories

Para cada RF essencial, defina critérios de aceite.

3) Priorizar o backlog (MoSCoW)

Must para o MVP; Could vira versão futura.

4) Preparar repositório e documentação

README + escopo + backlog inicial.
