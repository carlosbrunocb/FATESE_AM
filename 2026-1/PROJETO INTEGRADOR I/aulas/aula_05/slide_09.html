Boas práticas (anti-caos)

Commits

Pequenos e frequentes

Use mensagens claras (ex.: feat:, fix:).

PRs

Revisão rápida

1 pessoa revisa: evita bugs e alinha padrão.

Main

Sempre estável

Somente merge após validação.

Modelo simples de branches

main

Versão estável (o que será apresentado/entregue).

develop

Integração do time (onde o trabalho converge via PR).

feature/nome-curto

Uma funcionalidade por branch (ex.: feature/chat).

Regra de ouro

Nunca faça commit direto na main sem testar e sem PR.

Exemplo rápido (linha de comando)

git clone <url>

git checkout -b develop

git checkout -b feature/match

git add . && git commit -m "feat: match inicial"

git push -u origin feature/match

Depois, abra um Pull Request para develop e solicite revisão.

Por que isso importa no MVP?

Um fluxo Git simples reduz retrabalho, evita conflitos e acelera a entrega do recorte do MVP.
