O que entregar (postar no canal oficial)

1) URL do GitHub

Link do repositório oficial (público ou privado com acesso ao professor).

2) Lista refinada do MVP

Killer Feature + secundárias indispensáveis + recorte claro.

3) 3 RF (MVP)

Requisitos funcionais mínimos que viabilizam o fluxo principal.

4) 2 RNF (simplicidade)

Usabilidade, tempo de resposta, segurança mínima, clareza do fluxo.

Onde registrar

Coloque a lista do MVP no README e crie uma Issue por RF com critérios de aceitação.

Exemplo (como deve aparecer no README)

Killer Feature

Match por afinidade

Prova a proposta de valor.

Secundárias indispensáveis

Chat + agendamento + confirmação

Converte “match” em adoção real.

RNF de simplicidade

Contato em < 1 minuto

Foco no fluxo principal.

Checklist de qualidade do repositório

README completo

Descrição, escopo, como rodar e lista do MVP.

Branches organizadas

main (estável) e develop (trabalho do time).

Proteção da main

Merge via PR (revisão) — sem commit direto na main.

Colaboradores adicionados

Equipe + professor orientador com permissão.

Issues e (opcional) Project

1 Issue por RF com critério de aceitação.

Critério de “pronto” (DoD) — mínimo

A entrega está ok quando:
(1) o repo abre sem erro de permissão,
(2) README explica o MVP e como executar,
(3) existe pelo menos 1 PR (ou evidência de branches) e 1 Issue.

Por que isso importa?

Garante colaboração real, rastreabilidade e foco no MVP — sem “Espírito NASA”.

Dica rápida

Se o time ficar em dúvida, priorize: README + 3 RF + 2 RNF bem definidos e um fluxo simples demonstrável.
