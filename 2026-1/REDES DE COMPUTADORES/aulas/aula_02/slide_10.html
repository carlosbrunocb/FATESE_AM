Exemplo prático

O caminho do pacote (HTTP/HTTPS)

URL → DNS → TCP → TLS → HTTP

app → bits (e volta)

Cenário

Usuário digita www.exemplo.com.br e pressiona Enter. O navegador precisa encontrar o IP, abrir conexão e enviar a requisição.

Destino típico

:80

:443

Passo a passo (com protocolos)

Leia como uma linha do tempo. O dado desce encapsulando até virar bits.

1

DNS: nome → IP

UDP/53

(às vezes TCP)

O host vira um endereço IP. Sem isso, não existe “destino” para o IP levar o pacote.

Pergunta

Qual o IP de www.exemplo.com.br?

Resposta

IP do servidor (A/AAAA)

2

TCP: estabelecer conexão

SYN → SYN/ACK → ACK

Antes do HTTP, o TCP negocia parâmetros (MSS/janela), cria estado e garante entrega/ordem.

Porta origem

efêmera

Porta destino

80 ou 443

Bug clássico

porta errada

3

TLS (HTTPS): criptografar a sessão

certificado

chaves

Se for HTTPS, o HTTP fica “dentro” de TLS. O que aparece na rede é tráfego cifrado (com SNI, etc.).

Resultado

Cria uma chave de sessão → dados passam a ser cifrados e autenticados

4

HTTP: enviar a requisição

GET /index.html

Agora o app finalmente “fala”. No HTTP/1.1 é texto; no HTTP/2 é binário (frames).

Exemplo (simplificado)

GET /index.html HTTP/1.1
Host: www.exemplo.com.br
User-Agent: {browser}
Accept: */*

Encapsulamento (resumo final)

Mensagem HTTP → segmentos TCP → pacotes IP → quadros Ethernet/Wi‑Fi → bits no meio físico.

O que você vai “ver” no Wireshark

Portas TCP/UDP, IPs, MAC (na rede local), flags (SYN/ACK), e HTTP (se não estiver cifrado).
