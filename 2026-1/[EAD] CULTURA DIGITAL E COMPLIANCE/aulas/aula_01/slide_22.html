Segredos, pipeline e evidências

Case Study

Caso 1

Vazamento de

API Key

Cenário (realista)

Chave exposta no repositório

Erro de cultura/compliance

API Key commitada (ex.: .env, arquivo de config) e publicada em repositório público/compartilhado.

Falhas comuns associadas

Sem code review, sem scanner de segredos no CI, sem rotação e sem inventário de chaves.

Sinal de alerta: “Funciona local, então sobe assim mesmo” — segurança vira retrabalho.

Impacto (por que isso é grave?)

  • Comprometimento de contas e infraestrutura (cloud, APIs, banco).

  • Exfiltração de dados, indisponibilidade e aumento de custos (uso indevido).

  • Risco de incidentes e exigência de evidências (auditoria, LGPD).

Solução ideal

Controles técnicos + cultura de revisão

shift-left

1

Secrets Management

Usar Vault/KMS/Secrets do cloud e variáveis de ambiente (nunca no Git).

2

CI: Secret Scanning

Bloquear merges com scanner de segredos + políticas como código.

3

Rotação + resposta

Revogar/rotacionar chaves, revisar acessos e registrar evidências do incidente.

Diagrama

Pipeline com gate de compliance

Commit

pre-commit (secret hook)

CI Build

SAST/SCA + secret scan

Gate

falha → bloqueia merge

Deploy

segredos via Vault/KMS

Evidência de compliance: PR aprovado + logs do scanner + ticket de rotação + registro do incidente.

Lição aprendida

Segredo em Git não é “erro pequeno”: é incidente. Previna com automação + revisão + rotação.
