Scrum Essencial:

Papéis e Responsabilidades

No PI, o Scrum ajuda a evitar “todo mundo faz tudo” — e cria clareza para decidir, facilitar e executar.

Atenção (erro comum)

Papéis não são “chefia”. São responsabilidades para organizar decisões e remover travas do time.

Os 3 papéis (na prática do PI)

O que cada um garante para o trabalho avançar

Product Owner (PO)

Decide prioridades e define valor: o que entra primeiro no backlog do PI.

Entrega principal

Backlog priorizado

No GitHub

Issues + ordem

Scrum Master (SM)

Facilita o processo e remove impedimentos: ajuda o time a manter o fluxo e respeitar acordos (WIP/DoD).

Entrega principal

Ritmo + melhoria

No GitHub

Board “limpo”

Developers (Time)

Constrói e valida incrementos: implementa, testa, documenta e entrega com qualidade (DoD).

Entrega principal

Incremento pronto

No GitHub

Assignees + PR

Aplicação imediata no Projeto Integrador

O que “muda amanhã” no trabalho do grupo

PO define 5–10 itens mais valiosos

Cada item vira uma Issue com descrição e critérios de aceite.

SM protege o WIP

Se “In Progress” lotou, o time termina antes de puxar tarefa nova.

Devs assumem dono por Issue

Nada “órfão”: toda Issue tem Assignee e status real no board.

Regra simples para o PI

PO escolhe o que é mais importante → SM facilita o fluxo → Devs entregam com DoD.

Exemplo lúdico (memorização)

PO

Diretor(a)

Decide roteiro e prioridade das cenas (o que dá mais impacto).

SM

Produtor(a)

Resolve logística, remove travas e mantém o time sincronizado.

Devs

Equipe Técnica

Grava, edita, revisa e publica com “qualidade mínima” (DoD).

Dica: um aluno pode trocar de papel em outro ciclo, mas em cada semana o papel precisa estar explícito.
