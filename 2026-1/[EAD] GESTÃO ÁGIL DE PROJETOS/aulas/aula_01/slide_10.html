GitHub Projects

como “Scrumban” do time

O objetivo é transformar acordos de trabalho (Scrum + Kanban) em rotina visível: Issues + Board + responsáveis + critérios.

Regra de ouro

Issue bem escrita + WIP respeitado = menos retrabalho e mais entregas “Done”.

Conceitos Ágeis (Scrum + Kanban)

O que precisamos “ter claro” no time

Como pensamos

Backlog priorizado

Lista ordenada de trabalho. O PO ajuda a priorizar por valor e urgência.

Fluxo visual (Kanban)

Colunas que representam o trabalho: Backlog → To Do → In Progress → Review/Test → Done.

Limite de WIP

Começar menos para terminar mais. Regra prática: WIP ≈ nº de devs ± 1.

Políticas + DoD

Acordos explícitos: quando mover, o que é “pronto”, e critérios mínimos de qualidade.

Papéis e responsabilidade

Cada item tem dono (Assignee); Scrum Master ajuda a remover impedimentos.

Mini-exemplo (PI)

“Implementar login” vira Issue com critérios de aceite, responsável e passa no fluxo do board até “Done”.

No GitHub (como implementamos)

Onde cada coisa “mora” na ferramenta

Como executamos

Backlog → Issues

Título claro + descrição + critérios de aceite + assignee + labels (bug/enhancement).

Quadro Kanban → Projects (Board)

Colunas do fluxo + políticas (descrição do projeto) + arrastar cards para refletir o status real.

Responsável → Assignees

“Sem dono, sem movimento”. Toda Issue deve ter pelo menos 1 responsável.

Tipo/urgência → Labels

Ex.: bug, enhancement, documentation, urgent.

Entrega → Pull Requests + Commits

Vincular PR à Issue e fechar automaticamente (ex.: Fixes #12) garante rastreabilidade.
